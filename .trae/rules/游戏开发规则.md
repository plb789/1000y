## 一、项目概述
### 1.1 项目简介
《千年江湖》是基于经典千年武侠框架重构开发的**复古2D武侠MMORPG全栈网游**，保留原版经典江湖玩法、武学系统、门派体系、PK竞技、野外探索核心体验，同时完成架构现代化升级。
项目采用**Electron客户端（HTML5 Canvas前端）+ Go微服务（WebSocket）+ MySQL 8.0** 全栈技术体系，支持跨端客户端登录、多人在线实时联机、跨场景交互、数据实时统计，兼顾复古游戏体验与现代服务端稳定性、扩展性，可支持千人同时在线，适用于私服运营、局域网联机、定制化二次开发。

### 1.2 项目目标
1. 复刻经典千年核心武侠玩法，还原打怪、练功、武学、门派、交易、攻城等经典模块；
2. 摒弃老旧C++框架，基于Go+HTML5 Canvas重构全栈体系，降低开发与维护成本；
3. 搭建高可用分布式微服务架构，支持多服、分区、合服、热更新；
4. 提供规范源码结构，支持功能迭代、模组开发、皮肤/场景/道具自定义扩展。

### 1.3 适用人群
开发人员、运维人员、运营管理人员、二次开发开发者、游戏私服运维团队。

## 二、整体技术架构
### 2.1 架构总览
项目采用**微服务拆分 + 前后端分离**模式，整体分为：**Electron前端客户端层、Go微服务服务端层、配套工具链**，各服务通过网络协议解耦，独立部署、互不影响。

整体架构拓扑：
```
Electron客户端 <──WebSocket──> GatewayService(网关) <──HTTP──> LoginService / GameService
                                     │                              │
                                     │ Redis Pub/Sub                │ HTTP
                                     ▼                              ▼
                              跨网关广播(map:*频道)          DBService ──> MySQL / Redis
                                     ▲                              ▲
                                     │                              │
                              RegistryService(服务发现) ────────────┘
```

服务间通信：
- **前端 ↔ 网关**：WebSocket 长连接（二进制协议）
- **网关 ↔ 业务服务**：HTTP（Gin RESTful）
- **跨网关广播**：Redis Pub/Sub（订阅 `map:*` 频道）
- **服务注册/发现**：RegistryService（独立微服务，HTTP API）
- **异步消息总线**：RabbitMQ（支持 Kafka/HTTP 降级）

### 2.2 技术栈明细
#### 2.2.1 前端（Electron客户端）
- 核心技术：**原生HTML + Canvas 2D + JavaScript（无框架）**
- 运行载体：Electron（跨Windows/macOS/Linux，保留客户端体验；打包时由 `build_client.bat` 同步源码至 `Frontend/Electron` 目录构建）
- 渲染能力：Canvas 2D渲染引擎（原生解析千年官方 `.map` 二进制瓦片地图、DDS贴图、SPR帧动画、角色帧动画、场景特效）
- 网络通信：WebSocket长连接（二进制封包协议，实时联机、消息广播）
- 资源格式：兼容原版千年资源（DDS贴图、SPR帧动画、MAP场景地图、ATZ/ATD动画、EFT特效、PAL调色板、PGK打包）
- UI系统：**JSON配置驱动**的动态UI加载（`UIConfigManager` + `UIDynamicLoader`），配套可视化 `UIEditor` 设计工具
- 性能优化：瓦片地图分块加载（`ChunkManager`）、A*寻路、空间网格索引、轻量无框架设计
- 附加能力：全屏/窗口化切换、断线重连（Token会话保持）、本地缓存

#### 2.2.2 服务端技术栈（核心逻辑）
服务端采用 **Go 1.22 + Gin 框架**，拆分为 5 个独立微服务：

1. **GatewayService（网关微服务）**
   - 核心依赖：gorilla/websocket
   - 功能：客户端WebSocket接入、协议转发、连接管理、会话保持、多服路由、IP限制防攻击
   - 特色：按地图分桶索引（`clientsByMap`）加速同地图广播；Redis Pub/Sub 实现多网关实例跨实例广播（带 `SourceInstance` 防循环）
2. **LoginService（登录微服务）**
   - 功能：账号注册、登录校验、Token发放与验证、角色列表、角色创建
3. **GameService（游戏微服务，核心业务）**
   - 功能：角色、战斗、武学技能、道具、任务、地图、怪物AI、成就、BUFF、属性系统
   - 分布式特性：通过 `handled_maps` 配置按地图分片，多实例水平扩展；经 RegistryService 发现其他实例，支持跨服传送
   - 性能优化：`SpatialGrid` 空间网格索引将碰撞检测/视野查询从 O(N) 降至 O(1)
4. **DBService（数据微服务）**
   - 功能：MySQL CRUD封装、Redis缓存管理、数据持久化
   - 数据模型：账号、聊天、好友、公会、邮件、充值、签到、任务
5. **RegistryService（注册中心，服务发现）**
   - 功能：服务注册/注销、心跳检测、按地图/类型查询实例、自动清理过期实例（60秒周期）

#### 2.2.3 公共组件（Server/Common）
- `message_bus.go`：消息总线接口，支持 RabbitMQ / Kafka / HTTP 三种模式，带健康检查与自动恢复降级
- `distributed_router.go`：分布式路由，含负载均衡、故障转移、熔断器配置
- `registry.go`：服务注册中心客户端
- `cross_gateway_broadcast.go`：跨网关广播
- `monster_sync.go`：怪物同步
- `prometheus_metrics.go`：Prometheus 监控指标
- `config_loader.go`：JSON 配置加载（地图、NPC、怪物、武学、道具、掉落、BUFF、任务、成就、商店、公告、怪物生成点）
- `service_client.go` / `auth.go` / `timer.go` / `util.go`：服务客户端、鉴权、定时器、工具

#### 2.2.4 数据存储
- 主数据库：**MySQL 8.0** 存储角色、道具、账号、好友、公会、邮件、充值、签到、任务等持久化数据（最大连接100、空闲20）
- 缓存：**Redis 6+** 用于跨网关广播（Pub/Sub）、在线玩家缓存、临时状态、GameService 本地缓存（连接池100）

#### 2.2.5 配套工具链（Tools）
全部基于 **C# .NET 6 + WinForms** 开发，独立解决方案：
1. **资源编辑器（MillenniumResEditor）**：千年原版资源解析与导出，支持 DDS/SPR/ATZ/ATD/EFT/MAP/PAL/PGK 等格式查看、批量导出、地图编辑
2. **WebSocket调试工具（WsDebugTool）**：协议联调，支持连接管理、二进制封包收发、心跳、十六进制查看，完全对齐项目二进制协议
3. **前端UI设计生成工具（UIDesigner）**：可视化设计UI并生成 JSON 配置，供客户端 `UIConfigManager` 动态加载

### 2.3 网络协议设计
1. **前端 ↔ 服务端**：WebSocket长连接，**二进制封包协议**
   - 封包格式：`协议号(2字节 小端) + 数据长度(2字节 小端) + 消息体 + 校验码(1字节 累加和&0xFF)`
   - 协议号统一在 `Frontend/Protocol/ProtocolDefine.js` 与服务端定义对齐
2. **服务端内部**：HTTP（Gin RESTful）+ RabbitMQ 异步消息总线（可降级 Kafka/HTTP）
3. **跨网关广播**：Redis Pub/Sub（`map:*` 频道，多 Gateway 实例订阅，带来源实例ID防循环）

### 2.4 部署架构（标准生产环境）
```
外网 → 防火墙 → GatewayService(WebSocket:8080)
                    ↓ HTTP
        LoginService(8084) / GameService(8082)
                    ↓ HTTP
              DBService(8083) → MySQL / Redis
                    ↑
        RegistryService(8081) 服务发现
                    ↑
   GameService 多实例(按 handled_maps 分片) ←→ Redis Pub/Sub 跨网关广播
```

服务端口分配：
| 服务 | 端口 | 说明 |
|------|------|------|
| GatewayService | 8080 | WebSocket + HTTP |
| RegistryService | 8081 | HTTP API |
| GameService | 8082 | HTTP API |
| DBService | 8083 | HTTP API |
| LoginService | 8084 | HTTP API |

## 三、项目核心特色
### 3.1 玩法特色
1. **经典复古武侠体系**：完整复刻千年原版内功、外功、身法、护体武学修炼体系，支持手动练功、挂机练功、武学进阶。
2. **自由PK与江湖对抗**：野外自由PK、善恶值、红名机制、PK模式切换（和平/队伍/帮派/全体）。
3. **门派与社交系统**：门派、好友、组队、公会、聊天（私聊/队伍/世界/门派）。
4. **经济与交易系统**：货币体系（金币/绑定金币/元宝）、商店NPC、道具掉落、装备强化。
5. **大世界探索**：多张武侠场景、野外怪物、BOSS、地图传送、自动寻路（A*算法）。

### 3.2 技术架构特色
1. **全栈Go+HTML5重构**：摒弃老旧C++框架，基于Go微服务+HTML5 Canvas原生渲染，跨端兼容、维护成本低。
2. **WebSocket二进制协议**：低延迟长连接，封包含校验码防篡改，满足MMORPG实时联机需求。
3. **微服务解耦 + 服务发现**：网关、登录、游戏、数据、注册中心独立部署；RegistryService 动态发现，多 GameService 实例按地图分片水平扩展。
4. **空间网格索引**：`SpatialGrid` 将碰撞检测与视野查询复杂度从 O(N) 降至 O(1)，千玩家场景性能提升约100倍。
5. **跨网关广播**：Redis Pub/Sub + 按地图分桶索引，支持多 Gateway 实例横向扩展，带来源实例ID避免消息循环。
6. **消息总线降级机制**：RabbitMQ 异步通信，连接失败自动降级 HTTP，带健康检查与自动恢复。
7. **原生解析千年地图**：Canvas 2D 原生解析千年官方 `.map` 二进制瓦片地图，兼容原版资源（DDS/SPR/ATZ/ATD/EFT/PAL/PGK）。
8. **JSON配置驱动**：游戏数据（地图/NPC/怪物/武学/道具/掉落/BUFF/任务/成就/商店）全部 JSON 外置，无需重启即可热更。
9. **全链路监控**：Prometheus 指标采集，在线人数、服务器负载、异常连接全记录。
10. **安全防护**：WebSocket封包校验、IP连接数限制、Token会话、断线重连。

### 3.3 工具链特色
1. **资源编辑器**：一站式千年原版资源解析、查看、批量导出、地图编辑。
2. **WebSocket调试工具**：协议联调利器，二进制封包可视化收发。
3. **UI设计生成工具**：可视化设计UI，一键生成JSON配置，客户端动态加载。

- **Electron打包**：`build_client.bat` 将 `Game.js`/`index.html` 同步至 `Frontend/Electron` 目录后用 electron-packager 打包输出至 `Frontend/Build`

## 七、部署与运维
### 7.1 开发环境
1. **前端**：任意现代浏览器直接打开 `Frontend/index.html` 调试；Electron 打包需 Node.js + npm
2. **服务端**：Go 1.22+、MySQL 8.0、Redis 6+、RabbitMQ（可选，可降级HTTP）
3. **工具链**：.NET 6 SDK + Visual Studio（编译 C# 工具）

### 7.2 服务端启动
1. 数据库初始化：执行 `Script/SQL/init.sql`
2. 安装中间件：MySQL、Redis、（可选 RabbitMQ）
3. 启动顺序：**RegistryService → DBService → LoginService → GameService → GatewayService**
   - RegistryService 必须最先启动（服务发现基础）
   - GatewayService 最后启动（依赖 Login/Game 服务发现）
4. 一键启动：`Server/start_all.bat`（Windows）/ `Script/Shell/start.sh`（Linux）
5. 健康检查：`Server/check_services.bat`

### 7.3 客户端打包
1. 执行 `build_client.bat`
2. 脚本自动同步 `Game.js`/`index.html` 至 `Frontend/Electron`，npm install，electron-packager 打包
3. 输出至 `Frontend/Build/`

### 7.4 运维要点
- 守护进程保活各微服务；定时数据备份
- Prometheus 指标监控服务器负载、在线人数
- 日志按服务独立输出，便于排查

## 八、二次开发规范
1. 所有新增功能遵循**模块化开发**，不篡改核心底层代码；
2. 新增WebSocket协议统一在 `Frontend/Protocol/ProtocolDefine.js` 与服务端对齐定义，版本向后兼容；
3. 游戏数据全部外置到 `Server/GameService/Config/*.json`，禁止硬编码数值；
4. Canvas渲染新增特效/地图需兼容原版 `.map` 格式，保证加载性能；
5. Go微服务新增接口需遵循Gin RESTful规范，二进制封包格式 `[Cmd(2LE)][Len(2LE)][Body][Check(1)]` 保持统一；
6. 新增 GameService 实例需配置唯一 `instance_id` 与 `handled_maps`，并注册到 RegistryService；
7. 跨网关广播消息必须填充 `SourceInstance` 字段，避免循环广播；
8. 所有新增接口、功能必须编写日志与单元测试，便于问题排查。
