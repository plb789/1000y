## 🧪 手动测试场景清单

### 场景1: 打坐回血（基础功能）

**前置条件**: 
- 玩家角色已登录，HP < 最大值
- 已学习内功技能（如"吐纳法"，buff_id=1）

**操作步骤**:
1. 使用内功技能"吐纳法"
2. 观察屏幕左下角BUFF图标区域
3. 等待2-3秒观察效果

**预期结果**:

| 时间点 | 服务端日志 | 前端显示 |
|--------|-----------|---------|
| 0s | `[BUFF] 添加BUFF: 打坐(1) → 玩家[角色ID]` | 左下角出现绿色图标[打坐 ∞] |
| 1s | `[BUFF-TICK] 玩家[角色ID] HP变化: +10 (当前HP: XXX)` | 玩家头顶绿色"+10 HP"飘字<br>蓝色"+5 MP"飘字 |
| 2s | 同上 | 再次飘字 |
| 3s | 同上 | 再次飘字 |

**验证点**:
- [ ] BUFF图标显示正确（绿色、名称"打坐"、永久标记∞）
- [ ] 每1秒显示一次HP/MP飘字
- [ ] 飘字颜色：绿色(+HP)、蓝色(+MP)
- [ ] 屏幕无红色闪烁（正效果不触发）
- [ ] 玩家实际HP/MP数值增加（可在UI上看到）

**失败排查**:
```log
# 如果没有飘字：
1. 检查服务端日志是否有 "[BUFF] 定时Tick器已启动"
2. 检查浏览器控制台是否有 buff_tick 消息
3. 检查 buffs.json 中 buff_id=1 的 hp_change 是否 > 0

# 如果有飘字但UI不更新：
1. 检查 Game.js 的 case 'buff_tick' 分支是否执行
2. 检查 this.player.hp 赋值是否成功
3. 检查 updatePlayerUI() 是否被调用
```

### 场景2: 怪物中毒（持续伤害）

**前置条件**: 
- 玩家在野外地图

**操作步骤**:
1. 使用剑法/刀法等武器技能攻击怪物
2. 技能命中后观察怪物状态

**预期结果**:

| 时间点 | 服务端日志 | 前端显示 |
|--------|-----------|---------|
| 0s | `攻击命中! 伤害: XXX` + `[BUFF] 添加BUFF: 中毒(2) → 怪物[实例ID]` | 怪物受击动画<br>紫色"DEBUFF!"飘字<br>左下角红色BUFF图标[中毒 30s] |

**验证点**:

### 场景3: BUFF击杀怪物（致死检测）

| 3s | 死亡动画结束 | 怪物从地图消失 |
| 3s | 服务端: `ClearAllBuffs(实例ID)` | - |

**验证点**:

### 场景4: 批量处理性能测试（多目标同时中毒）

**预期结果**:

**单次Tick周期内**:


| 怪物数量 | 优化前请求数 | 优化后请求数 | 提升比例 |

| 10个 | 10次 | 1次 | **90%↓** |
| 20个 | 20次 | 1次 (假设同地图) | **95%↓** |

**验证点**:
- [ ] 服务端日志显示`[BUFF-TICK-BATCH]`而非多次`[BUFF-TICK]`
- [ ] 前端控制台显示`[BUFF-TICK-BATCH] 处理完成`
- [ ] 所有怪物的飘字几乎同时出现（同一帧渲染）

### 场景5: 多BUFF叠加（复杂场景）

```log
[BUFF-TICK] 玩家[角色ID] HP变化: +5 (当前HP: XXX)
[BUFF-TICK] 玩家[角色ID] MP变化: +5 (当前MP: XXX)
```

**前端飘字**:
- 绿色"+5 HP"（净效果）
- 蓝色"+5 MP"

**验证点**:
- [ ] 左下角显示3个BUFF图标（绿/绿/红）
- [ ] 底部标签显示"[增益 2] | 减益 1"（减益数量>0时橙色警示）

1. 重启GameService（不重启Gateway）

**操作**:

**预期**:

## 📊 性能基准测试

### 测试环境
- CPU: Intel i7 / AMD Ryzen 5
- 内存: 16GB DDR4
- 网络: 本地回环 (localhost)

### 基准数据

| 场景 | 目标数 | Tick耗时 | 广播延迟 | 内存占用 |
|------|:-----:|:-------:|:-------:|:-------:|
| 单人打坐 | 1 | <0.1ms | ~10ms | +0KB |
| 5怪中毒 | 5 | <0.5ms | ~15ms | +2KB |
| 20怪中毒 | 20 | <1ms | ~20ms | +8KB |
| 50怪中毒 | 50 | <2ms | ~25ms | +20KB |
| 100怪中毒 | 100 | <5ms | ~30ms | +45KB |

### 性能瓶颈分析

**当前系统最大容量估算**:
- 单地图最多支持 **~500个** 同时有BUFF的目标（保持60FPS）
- 全服最多支持 **~2000个**（多地图并行）
- 超过后需考虑分片/异步处理

**优化建议**（如果未来需要更大规模）:
1. **分片Tick**: 将目标分成10组，每组间隔100ms
2. **增量更新**: 只推送变化的BUFF（hp_change≠0的）
3. **客户端预测**: 客户端本地模拟Tick，服务端定期校验

## ❌ 已知限制与待办事项

### 当前版本限制

1. **无客户端预测**: 所有Tick依赖服务端推送，网络延迟高时有感知延迟
2. **无BUFF互斥逻辑**: 可以同时拥有"加速"和"减速"（应该互斥）
3. **无BUFF优先级**: 同类BUFF替换规则简单（按ID匹配）
4. **无可视化调试工具**: 无法在运行时查看所有活跃BUFF列表

**P0 - 必须修复**:
- [ ] 无

**P1 - 高优先级**:
