# 服务端架构说明

本文档详细说明《千年江湖》服务端架构、各服务功能、数据流向及并发能力评估。

## 二、服务功能详解

### 1. LoginService（登录服务）

| 功能 | 方法/接口 | 说明 |
|------|----------|------|
| 用户登录 | `POST /api/auth/login` | 验证账号密码，返回Token |
| 用户注册 | `POST /api/auth/register` | 创建新账号 |
| Token验证 | `POST /api/auth/verify` | 验证Token有效性 |
| 获取角色列表 | `POST /api/role/list` | 获取玩家角色列表 |
| 创建角色 | `POST /api/role/create` | 创建新角色 |

### 2. GatewayService（网关服务）

| 功能 | 方法/接口 | 说明 |
|------|----------|------|
| WebSocket连接 | `ws://localhost:8080/ws` | 客户端连接入口 |
| 消息转发 | `forwardMoveToGameService()` | 转发移动消息到GameService |
| 跨网关广播 | `PublishToMap()` | 通过Redis发布地图消息 |
| 订阅频道 | `subscribeToMapChannels()` | 订阅map:*频道 |
| 内部推送 | `POST /internal/push` | GameService推送消息给玩家 |
| 内部广播 | `POST /internal/broadcast` | GameService广播消息 |
| 内部广播地图 | `POST /internal/broadcast_map` | GameService广播地图移动消息 |

**Redis 用途：**
- 跨Gateway实例的消息广播（Pub/Sub模式）
- 订阅 `map:*` 频道，同步玩家移动、进入/离开、聊天等事件
- 支持水平扩展，增加Gateway实例无需修改配置

### 3. GameService（游戏逻辑服务）

| 模块 | 功能 | 接口 |
|------|------|------|
| **Role** | 角色管理 | `/api/role/*` |
| **Map** | 地图管理/移动 | `/api/map/move`, `/api/map/teleport` |
| **Battle** | 战斗系统 | `/api/battle/attack`, `/api/battle/skill` |
| **Skill** | 技能系统 | `/api/skill/cast`, `/api/skill/upgrade` |
| **Item** | 道具系统 | `/api/item/use`, `/api/item/equip` |
| **Quest** | 任务系统 | `/api/quest/accept`, `/api/quest/complete` |

**分布式特性：**
- 通过 `handled_maps` 配置处理特定地图
- 多实例可分别处理不同地图，实现水平扩展
- 通过RegistryService发现其他GameService实例
- 支持跨服传送（玩家数据跨实例传递）

### 4. DBService（数据库服务）

| 功能 | 接口 | 说明 |
|------|------|------|
| 角色数据 | `/api/role/*` | 角色CRUD操作 |
| 道具数据 | `/api/item/*` | 道具背包管理 |
| 好友数据 | `/api/friend/*` | 好友关系管理 |
| 邮件数据 | `/api/mail/*` | 邮件系统 |
| 充值数据 | `/api/recharge/*` | 充值记录 |
| 聊天数据 | `/api/chat/*` | 聊天消息存储 |

**数据库配置：**
- MySQL：最大连接数100，空闲连接20
- Redis：连接池50，最小空闲5

### 5. RegistryService（注册中心）

| 功能 | 接口 | 说明 |
|------|------|------|
| 服务注册 | `POST /api/registry/register` | 服务实例注册 |
| 服务注销 | `POST /api/registry/unregister` | 服务实例注销 |
| 心跳检测 | `POST /api/registry/heartbeat` | 保持服务存活 |
| 获取实例列表 | `GET /api/registry/list` | 获取所有服务实例 |
| 按地图获取 | `GET /api/registry/map/{map_id}` | 获取处理指定地图的GameService |
| 按类型获取 | `GET /api/registry/type/{service_type}` | 按服务类型获取实例 |

**设计特点：**
- 独立服务，不依赖其他服务
- 内存存储 + 可选Redis持久化
- 自动清理过期实例（60秒周期）
- 支持服务降级（HTTP/gRPC备用通道）

## 三、核心数据流

### 3.1 用户登录流程

```
Client → Gateway → LoginService → DBService → LoginService → Gateway → Client
```

1. 客户端通过WebSocket连接Gateway
2. Gateway转发登录请求到LoginService
3. LoginService验证账号密码（查询DBService）
4. 返回Token给客户端
5. 客户端使用Token获取角色列表

### 3.2 玩家移动流程

```
Client → Gateway → GameService(验证位置) → DBService(更新位置) 
  → GameService → Gateway(广播) → 同地图玩家
```

1. 客户端发送移动消息到Gateway
2. Gateway转发到GameService处理移动逻辑
3. GameService验证位置合法性（碰撞检测）
4. GameService更新数据库中的玩家位置
5. GameService调用Gateway的内部广播接口
6. Gateway广播移动消息给同地图所有玩家

### 3.3 跨服传送流程

```
Client → Gateway → GameServiceA → RegistryService(查目标) 
  → GameServiceB → Gateway → Client
```

1. 玩家触发传送（如进入传送门）
2. GameServiceA查询RegistryService获取目标地图的GameServiceB
3. GameServiceA将玩家数据传递给GameServiceB
4. GameServiceB加载玩家到新地图
5. Gateway更新玩家的地图ID
6. 广播玩家进入新地图

### 3.4 跨网关广播流程

```
Gateway A → Redis (Publish map:1) → Gateway B/C/D (Subscribe map:*)
```

1. 玩家在Gateway A上移动
2. Gateway A将移动消息发布到Redis的 `map:1` 频道
3. 所有Gateway实例都订阅了 `map:*` 模式
4. Gateway B/C/D收到消息后转发给自己的客户端
5. 同一地图的玩家都能看到移动消息

## 四、并发能力评估

### 4.1 当前配置瓶颈分析

| 服务 | 当前配置 | 理论瓶颈 | 实际瓶颈 |
|------|----------|----------|----------|
| **GatewayService** | 无硬编码限制 | 无限 | Redis Pub/Sub带宽 |
| **GameService** | 可分布式部署 | 无限 | 单实例CPU/内存 |
| **DBService** | MySQL连接100 | ~100并发DB操作 | MySQL吞吐量 |
| **Redis** | 连接池50 | ~50并发操作 | Redis单线程瓶颈 |
| **RegistryService** | 无限制 | 无限 | 心跳检测频率 |

### 4.2 理论并发上限估算

#### WebSocket连接数（GatewayService）

```yaml
# 当前配置
send_channel_size: 512          # 每玩家缓冲512条消息
broadcast_channel_size: 5000    # 广播队列5000条
max_connections_per_ip: 10      # 同IP限制（防攻击）
```

- **理论上限**：无硬编码限制
- Go的WebSocket使用goroutine，单机可支持 **10万+ 连接**
- 实际瓶颈：系统文件描述符限制（ulimit）、内存

#### 数据库并发（DBService）

```yaml
# 当前配置
max_open_conns: 100    # MySQL最大连接数
max_idle_conns: 20     # 空闲连接池
```

- 100个MySQL连接 → 每次DB操作需等待连接
- 假设每次操作50ms → **2000次/秒**
- 玩家登录/移动/战斗都需DB操作 → **约2000玩家活跃操作**

#### Redis并发

```yaml
# 当前配置
pool_size: 50          # 连接池大小
```

- Redis单线程，但IO多路复用
- 50连接池 → **5000-10000次/秒**（取决于操作复杂度）
- 主要用于：跨网关广播、缓存读写

#### GameService逻辑处理

- Go HTTP服务单机可处理 **1万+ QPS**
- 战斗/移动逻辑计算 → **CPU密集型**
- 单实例估算：**5000-10000玩家同时活跃**

### 4.3 综合并发能力

#### 单实例部署（当前配置）

| 场景 | 预估并发 | 瓶颈点 |
|------|----------|--------|
| **在线玩家数** | 5,000-10,000 | DBService MySQL连接 |
| **同时活跃玩家** | 2,000-3,000 | DB写入吞吐 |
| **每秒移动请求** | 5,000+ | GameService CPU |
| **每秒战斗请求** | 1,000-2,000 | GameService CPU |
| **跨网关广播** | 10,000+ | Redis带宽 |

#### 分布式部署（扩展后）

| 扩展方案 | 预估并发 |
|----------|----------|
| **当前配置** | 5,000-10,000在线 |
| **DB读写分离** | 20,000-30,000在线 |
| **Redis Cluster** | 50,000+广播吞吐 |
| **GameService分片** | 无限（按地图） |
| **Gateway多实例** | 10万+连接 |

### 4.4 结论

**当前架构（不限制硬件）理论并发上限**：

| 指标 | 估算值 |
|------|--------|
| **最大在线人数** | **10万+**（Gateway无限制） |
| **同时活跃玩家** | **2,000-5,000**（DB瓶颈） |
| **每秒请求处理** | **5,000-10,000 QPS** |

**主要瓶颈**：DBService的MySQL连接数（100），这是最容易扩展的部分。只需增加MySQL连接数或部署读写分离，即可大幅提升并发能力。

## 五、关键技术特性

| 特性 | 实现方式 |
|------|----------|
| **分布式部署** | 多GameService实例处理不同地图 |
| **服务发现** | RegistryService维护服务-地图映射 |
| **跨网关广播** | Redis Pub/Sub模式 |
| **负载均衡** | 根据地图分配到不同GameService |
| **高可用** | 服务心跳检测，自动剔除失效实例 |
| **服务降级** | 核心业务采用主备通道设计 |
| **消息合并** | 移动消息可配置合并延迟，减少广播量 |
| **视野范围** | 可配置视野范围（默认500像素），只广播视野内玩家 |

## 六、配置文件说明

### 6.1 服务端口配置

| 服务 | 端口 | 说明 |
|------|------|------|
| GatewayService | 8080 | WebSocket + HTTP |
| RegistryService | 8081 | HTTP API |
| GameService | 8082 | HTTP API |
| DBService | 8083 | HTTP API |
| LoginService | 8084 | HTTP API |

### 6.2 服务依赖关系

```
LoginService ──────→ DBService
   (只依赖)

GatewayService ───→ LoginService (登录/注册)
                ───→ RegistryService (服务发现)
                ───→ Redis (跨网关广播)

GameService ────────→ DBService (数据持久化)
                 ───→ RegistryService (服务注册)
                 ───→ GatewayService (推送消息)

RegistryService ────→ (无外部依赖)
```

### 6.3 配置文件说明

| 服务 | 配置文件 | 实际使用的服务地址 |
|------|----------|-------------------|
| **LoginService** | Login.yaml | db_service |
| **GatewayService** | Gateway.yaml | login_service, registry_service |
| **GameService** | Game.yaml | db_service, gateway_service, registry_service |
| **DBService** | DB.yaml | (无外部依赖) |

**注意**：
- 每个服务只配置自己实际需要的下游服务地址
- 服务发现通过 RegistryService 实现动态查找，无需硬编码
- 多 GameService 实例时，Gateway 通过 RegistryService 动态发现，无需配置多个地址

## 七、扩展建议

### 7.1 当前架构瓶颈

| 瓶颈 | 原因 | 解决方案 |
|------|------|----------|
| **DBService MySQL** | 单实例100连接 | 读写分离、分库分表 |
| **Redis单实例** | Pub/Sub带宽限制 | Redis Cluster |
| **GameService单实例** | CPU限制 | 按地图分片部署多实例 |

### 7.2 推荐扩展路径

1. **阶段一**：增加MySQL连接数（100→500），支持5万在线
2. **阶段二**：部署MySQL读写分离，支持10万在线
3. **阶段三**：Redis Cluster + GameService分片，支持50万+在线
4. **阶段四**：Gateway无状态化 + Kubernetes编排，支持百万在线
